# 2.5 Data export + public query API

**Tier:** 2 (differentiator) · **Status:** planned

Two surfaces sharing one auth model. Reuse Bearer API-key auth for the public/CLI API; session scope for the in-dashboard "Export" button.

## Approach

- **Middleware:** add `isQueryApiRoute(path)` to `src/lib/ingest-routes.ts` + a branch in `02.api-auth.ts` (or `02b`) doing Bearer lookup + `getApiKey` stash **without** `negotiateVersionOrResponse` and under a looser rate-limit (query ≠ ingest). Keep the IP backstop.
- **`routes/api/v1/*`** — `runs/index.ts` (list with branch/env/status/date/cursor filters, reuse `runs-filters-where.ts` + `encodeCursor`/`decodeCursor`), `runs/[runId]/tests.ts` (reuse `loadRunResultsPage`), `runs/[runId].ts` (reuse `summary.ts` columns). Scope everything with `runScopeWhere(scope)`.
- **Export:** `?format=csv` (or `routes/api/v1/export/*`) with a small pure `src/lib/csv.ts` (RFC-escaped, unit-tested), streamed cursor-paged with a `WRIGHTFUL_EXPORT_MAX_ROWS` cap. In-dashboard export reuses the same query + serializer under session-authed `/api/t/.../export/*`.
- **Docs:** an endpoints/auth/cursor/CSV-columns page linked from the API-key management UI. (Optional follow-up: `apiKeys.scopes` to split read-only vs ingest keys.)

## Verification

- Unit: `csv.ts` escaping + filter-WHERE.
- Integration: valid key → scoped 200; no/invalid key → 401 (**not** the version-409 path, proving the branch is separate); project-A key can't see B; `?format=csv` well-formed.
- Manual: `curl` + open CSV in a spreadsheet.
