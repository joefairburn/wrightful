# 3.1 Granular RBAC + member-role editing

**Tier:** 3 (enterprise — defer until a paying customer pulls) · **Status:** planned

Widen `MembershipRole` to `"owner" | "member" | "viewer"`. The column is already `text` + `.$type<>()`, so union widening is the only schema change (zero migration) — this resolves the "no schema backing" worklog note.

## Approach

- A `can(role, action)` capability helper (`src/lib/roles.ts`) with actions `viewSettings | manageMembers | mintKeys | writeConfig | deleteTeam`.
- Generalize `gateTeamScope` to take a capability; add `requireRoleScope(c, action)` alongside the existing gates.
- `updateMemberRole` action on `members.server.ts` reusing the **race-safe last-owner guard** from `leaveTeam` (owner-count subquery in the UPDATE WHERE, not check-then-write).
- Member-remove UI (server action already exists).
- Invite POST accepts a validated `role` (currently hardcoded `"member"`).
- Viewers are members for read paths — no realtime/read changes.

## Verification

- Unit: `can()` matrix + last-owner demotion guard.
- Manual: viewer reads dashboard but 404s on owner-only settings, can't mint keys.
