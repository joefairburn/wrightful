# 2.6 TCP/ping monitors + deferred uptime phases

**Tier:** 2 (differentiator) · **Status:** planned

## Framing correction

**HTTP uptime monitoring is already complete and wired end-to-end** — executor, schemas, form, `?type=http` chooser, create action + cap, `uptime` queue with batching/retries, scheduler routing, detail-page windows + response-time trend, `statusCode`/`resultDetail` columns. It is _not_ a gap. The genuine remaining monitoring work is below.

## TCP/ping checks

(`"tcp"|"ping"` already reserved in types.) New executor `src/lib/monitors/tcp/tcp-run.ts` using `connect()` from `cloudflare:sockets`. Workers can't send ICMP — model "ping" as TCP-connect-to-port or HTTP HEAD; **document this**. Mirror the HTTP slice exactly:

- Config schema + URL-policy host blocking
- `tcpConfigFromForm` form-parse + `TcpMonitorForm` + a 3rd `TypeCard` in the chooser
- create/update action branch + `WRIGHTFUL_TCP_MONITOR_MAX_PER_PROJECT` cap
- `resolveExecutor` branch routed to the `uptime` queue
- `TcpResultDetail` type + Zod mirror + rendering

## Deferred Phases 4–6

- **Sub-minute scheduling:** Cloudflare cron floor is 1 min → fan-out N delayed jobs with `delaySeconds` offsets in `planMonitorSweep` (e.g. a 15s monitor → 4 jobs at 0/15/30/45s). Presets already exist in `HTTP_INTERVAL_PRESETS`, just hidden behind the v1 subset. Plan is pure/unit-testable.
- **In-executor retries:** the `retryConfig` column + `MonitorExecution.attempt` already exist but are unused — retry inside `runHttpCheck`/`runTcpCheck` before settling `fail`, bounded by the hard-timeout.
- **Alerting hooks:** at the single settle point `recordExecutionResult` (tracked under the separate alerting effort).
