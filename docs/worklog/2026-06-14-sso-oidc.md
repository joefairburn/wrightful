# 2026-06-14 — SSO / OIDC (roadmap 3.3): inert scaffolding + org-mapping v1

## Summary

Implemented the complete, correct, **inert-by-default** scaffolding for enterprise
SSO/OIDC (roadmap 3.3): the env keys, the `ssoEnabled` request-time predicate, an
owner-configured email-domain → team org-mapping (schema column + pure resolver +
DB seam), the login/signup "Continue with SSO" affordance, and the team-settings
SSO-domain card. The one remaining step — registering the Better Auth SSO plugin in
`auth.ts` — is **NOT wired**, by deliberate decision (see below). It is left as a
one-step, clearly-documented follow-up. No dependency was added.

**The #1 success criterion holds:** with no SSO env set (the clean-checkout default),
the app boots (`void prepare` ✓), typechecks (✓), and behaves **exactly** as before
— email/password + optional GitHub. SSO is fully inert: no plugin, no button, no
behavior change. Confirmed explicitly.

## Decision: inert scaffolding, NOT a plugin wire — and why

The decision rule in the brief: wire the plugin IF Void cleanly supports it AND a
compatible `@better-auth/sso` can be added without fighting the pinned toolchain;
otherwise build inert scaffolding and document the blocked wire. I took the
**inert-scaffolding** path. The investigation (verifying the recon, not redoing it):

1. **Void DOES expose the plugin slot.** `defineAuth`'s config type is
   `VoidAuthConfig = BetterAuthOptions | ((ctx) => BetterAuthOptions)`
   (`node_modules/.../void/dist/runtime/auth.d.mts` → `auth-CDeb2XAl.d.mts:18`), and
   `@better-auth/core`'s options declare `plugins?: ([] | BetterAuthPlugin[]) | undefined`
   (`@better-auth/core/dist/types/init-options.d.mts:682`). So registering `sso()` in
   the returned config is the intended shape — the slot is not the blocker.

2. **`@better-auth/sso` is NOT bundled in the pinned better-auth 1.6.11.** Its
   `package.json#exports` has `./plugins/generic-oauth` and `./plugins/oidc-provider`
   (which makes Wrightful an OIDC _provider_, not a _consumer_) but **no
   `./plugins/sso`**. `@better-auth/sso` is a separate npm package, not installed
   (`better-auth` itself is transitive via `void@0.9.2`, not a direct dep).

3. **A version-matched `@better-auth/sso@1.6.11` exists and its peers DO match**
   Void's bundled stack cleanly (`better-auth ^1.6.11`, `@better-auth/core ^1.6.11`,
   `@better-auth/utils 0.4.0`, `@better-fetch/fetch 1.1.21`, `better-call 1.3.5` —
   all match what's installed). So the _version_ fear is resolvable by pinning to
   1.6.11. (`@better-auth/sso@latest` is 1.6.18 and demands `better-auth ^1.6.18`,
   which would fork a SECOND copy of better-auth and register the plugin against the
   wrong instance — that one must NOT be used.)

4. **BUT its runtime deps are Cloudflare-Workers-hostile.** `@better-auth/sso@1.6.11`
   depends on `samlify@~2.10.2`, which pulls `xml-crypto`, `@xmldom/xmldom`,
   `node-rsa`, `node-forge`, `xpath` — a Node-native XML/crypto stack. The deploy
   target is Workers (workerd). Importing the SSO plugin barrel drags `samlify` into
   the Worker bundle **even for OIDC-only use** (SAML support isn't tree-shaken out).
   This is a real destabilization risk that **cannot be exercised here** (no IdP, no
   deploy), which is exactly the case the brief says not to force.

Per the decision rule ("incompatible / would destabilize auth / fighting the
toolchain → do NOT force it"), I did not install the plugin. Forcing it would have
risked the working auth system to satisfy a feature that can't be end-to-end tested
in this environment — the explicit "safety first" failure mode.

### Design choices worth recording

- **No dependency added.** `package.json`, `pnpm-lock.yaml`, and `pnpm-workspace.yaml`
  are untouched — the pinned vite-plus catalog is intact.
- **Sub-domain handling (documented choice):** `extractEmailDomain` does NOT collapse
  sub-domains. `user@eng.acme.com` yields `eng.acme.com`, which matches only a team
  that claimed exactly `eng.acme.com` — NOT one that claimed `acme.com`. A parent-domain
  claim auto-joining every sub-domain is a privilege-escalation footgun; a deployment
  that wants sub-domain inheritance claims each sub-domain explicitly.
- **`SsoDomainParse` is a discriminated result** (`{ ok: true; domain: string | null } | { ok: false }`),
  NOT `string | null | "invalid"` — the `"invalid"` literal would be subsumed by
  `string` and trip oxlint's `no-redundant-type-constituents`. (The `parseDays` sibling
  in `general.server.ts` can use `number | null | "invalid"` because `number` and the
  literal are disjoint; for strings they aren't.)
- **`joinTeamForSsoEmail` inserts a `member` membership** (not owner) into the claimed
  team, idempotently (composite `(userId, teamId)` read first). It is the runtime entry
  point the sign-in hook will call once the plugin is wired — intentionally unreferenced
  today, so it adds zero behavior to the live flows.
- **`updateSso` records no audit row**, matching its closest sibling `updateRetention`
  (also no audit). Kept symmetric to avoid scope creep; an audit action can be added
  later if settings edits become auditable across the board.

## What remains / how to finish the IdP wire

The wire is one step, gated on finding a **Workers-compatible OIDC plugin path**. To
finish (also documented inline in `auth.ts`):

1. **Pick a Workers-safe OIDC plugin.** Options, in preference order:
   - A vetted `@better-auth/sso` release whose runtime deps (currently `samlify` +
     a Node-native XML/crypto stack) bundle cleanly on workerd — verify by building
     the Worker and confirming no Node-builtin / native-binding bundling errors. Pin
     it to a version whose peers match Void's bundled better-auth (the 1.6.11 line
     peer-matches today; re-verify on the chosen version).
   - OR `genericOAuth` from better-auth's already-bundled `./plugins/generic-oauth`
     (no extra dep, no `samlify`), configured against the IdP's OIDC discovery
     document. This is the lower-risk path for OIDC-only and avoids the SAML baggage
     entirely; SAML would then be a later, separate addition.
2. **Register it conditionally in `auth.ts`**, mirroring the `socialProviders.github`
   gate: inline the SSO creds the same config-time way the GitHub creds are read
   (`process.env.SSO_*` — the `@/lib` alias isn't resolvable in the bare-Node
   `void prepare` context), gate on all-three-present, and append:
   `plugins: [...(defaults.plugins ?? []), ...(ssoConfigured ? [oidcPlugin({ issuer: SSO_ISSUER_URL, clientId: SSO_CLIENT_ID, clientSecret: SSO_CLIENT_SECRET })] : [])]`.
   Keep it conditional so the no-SSO path stays byte-identical.
3. **Call `joinTeamForSsoEmail(user.id, user.email)`** from the SSO sign-in /
   account-create hook (alongside the existing GitHub-mirror hook) so a verified SSO
   email auto-resolves into the team that claimed its domain. Only verified emails
   should drive this (the directed-invite seam already treats unverified emails as
   non-identities — apply the same rule).
4. **Add the SSO client route** if needed: the login/signup buttons point at
   `/api/auth/sign-in/sso`, Better Auth's SSO sign-in convention — confirm the chosen
   plugin mounts that (or adjust the button `href`).
5. **End-to-end test against a local OIDC test IdP** (per the roadmap verification
   line): sign-in creates a session and resolves into the expected team.

SAML remains a documented follow-up on the same plugin once OIDC is proven.
