### Why `@cloudflare/vite-plugin` stayed at 1.38.0

It is a transitive dependency of `void` (`^1.36.3`), not a direct dep. pnpm kept
the already-resolved 1.38.0 (which satisfies the range), so its existing dedup
patch applies verbatim — only its `patchedDependencies` entry needed re-adding.
Forcing 1.43.0 would require an `overrides` entry _and_ re-authoring the patch
(the un-deduped `return defu(...)` is still present in 1.43.0), for no functional
gain while stepping outside void 0.10.4's declared compat range. Left as-is.

### Not a Vite-major migration

Both `void@0.9.2` and `void@0.10.4` peer on `vite: ^8.0.0`, and this repo already
satisfies that (the catalog aliases `vite` → `@voidzero-dev/vite-plus-core`, the
Vite+ fork). So the jump was a minor Void bump, not a Vite 7→8 migration.

## The patches — none fixed upstream in 0.10.4

All three patches address bugs still present in 0.10.4 (verified by diffing the
0.10.4 dist), so all three remain necessary. Because pnpm patches are keyed to an
exact version, each was re-generated via `pnpm patch <pkg>@0.10.4` → edit → `pnpm
patch-commit`.

1. **`patches/void@0.10.4.patch`** (was `void@0.9.2.patch`) — two hunks:
   - `dist/index.mjs` `getInstance()`: fresh `pg.Pool` per query chain for BOTH
     dev and prod (removes the `_prodInstance` module-scope cache that hangs every
     authed prod request on workerd — see
     `2026-06-24-fix-void-pg-cross-request-pool-hang.md`). **Still shipped
     unpatched in 0.10.4.**
   - `dist/pages/protocol.mjs` `streamDeferredResponse()`: adds `X-VoidPages` +
     `Vary` headers for pages (SPA-nav) requests so the client router doesn't fall
     back to a full page load on deferred responses. **Still missing in 0.10.4.**

2. **`patches/@void__react@0.10.4.patch`** (was `@void__react@0.9.2.patch`) — the
   deferred-hydration `useId` fix. Server (`pages-server.mjs`) keeps the
   `__VOID_DEFERRED_SHELL_END__` `<template>` marker in the streamed HTML; client
   (`pages-client.mjs`) reproduces the `<Fragment>{page}<template/></Fragment>`
   root shape when the marker is present, so client `useId` matches the server and
   Base UI ids don't desync on hydration. **Still broken in 0.10.4.**
   - Note: 0.10.4 restructured the client hydration path (added a
     `data-void-ssr === "false"` → `createRoot(...)` branch). The Fragment wrap is
     applied only to the `hydrateRoot` branch (the `createRoot` branch is a fresh
     client render with no server HTML to match), so the port is not a mechanical
     line-shift of the 0.9.2 patch.

3. **`patches/@cloudflare__vite-plugin@1.38.0.patch`** — unchanged (version held).
   De-dupes `compatibility_flags` after `customizeWorkerConfig` merges, preventing
   the duplicate-`nodejs_compat` error. Only its `patchedDependencies` entry was
   re-added.

Deleted: `patches/void@0.9.2.patch` (tracked) and `patches/@void__react@0.9.2.patch`
(untracked WIP from this branch).
