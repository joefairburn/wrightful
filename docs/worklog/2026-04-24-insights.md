## Recurring technical decisions (apply to all four pages)

**O(buckets), not O(runs).** Every chart is backed by a single
aggregated SQL query that returns at most one row per rendered
bucket — a 90d daily view returns ≤90 rows regardless of whether
the project ran 500 or 500k tests. Run Duration initially pulled
raw `(createdAt, durationMs)` per run and was refactored to SQL-side
percentile picking during review.

**Discrete percentiles via window functions.** SQLite has no
`percentile_cont`, so we rank rows inside a CTE:

```sql
row_number() OVER (PARTITION BY bucket ORDER BY durationMs) AS rn,
count(*)     OVER (PARTITION BY bucket)                    AS cnt
```

and pick the target row with
`MIN(CASE WHEN rn = MAX(1, CAST(ROUND(cnt * q) AS INTEGER)) THEN durationMs END)`.
The `MAX(1, …)` floor keeps single-row buckets from resolving to
rank 0. Trade-off: no R-7 interpolation — imperceptible at chart
resolution and much simpler than emulating it in SQLite. This
pattern is reused on Slowest Tests (per-testId p95) with a second
window ranking by `createdAt DESC` to pull the _latest_ title / file
/ status through the same `GROUP BY`.

**DO-SQLite text-affinity gotchas.** Two pitfalls surfaced on
Run Status and are now codified in `bucketExpr`'s JSDoc so every
downstream page gets them for free:

1. **Literal divisors.** `sql\`createdAt / ${DAY_SEC}\``binds 86400
as a parameter; the driver applies text affinity, and`/` turns
into string concatenation — day/week buckets silently return
zero rows. Fix: inline literals (`/ 86400`) or use
`sql.raw(String(n))` where the divisor is computed.
2. **Group by the expression, not the alias.** `.groupBy("bucket")`
   doesn't consistently resolve the aliased expression. Pass
   `bucketExpr` directly.

**RSC-first charts, HTML tooltips.** All charts render inside RSC
pages — `@visx/scale` + `@visx/shape` primitives with a
non-uniform `preserveAspectRatio="none"` viewBox so strokes scale
fluidly with container width. Tooltips are HTML overlays driven by
CSS `group-hover`; no client JS is mounted for the charts
themselves. The only client islands on these pages are the branch
filter combobox (reused from `RunHistoryBranchFilter`) and shared
`ui/` primitives that happen to be client-boundary.

**KPI denominator convention.** Pass- and flaky-rate denominators
exclude `skipped` — matching the rule used on `flaky-tests.tsx` so
a flaky number in one place means the same thing everywhere.

rather than shipped as fake/guessed data:

| Prototype                                          | Outcome                                    | Why                                                                                        |

| Suite-density warning (E2E ratio)                  | **Dropped**                                | Depends on test-type classification.                                                       |

| Export button                                      | **Dropped**                                | Feature creep.                                                                             |

## Notable per-page details

- **Run Status** — four-segment stacked bar (passed/failed/flaky/
  skipped); KPIs use `sum(passed)` / `sum(failed)` etc. grouped per
  bucket. This was the first page and kickstarted the literal-
  divisor + groupBy-expression lessons above.
- **Suite Size** — supports an `all` range; to avoid rendering
  decades of empty leading shells, a small `min(createdAt)` query
  anchors the shell skeleton. Top-N spec files + top-N tags use
  `count(distinct testId)` so a prolific file doesn't win by
  volume of runs.
- **Run Duration** — segment defaults to `week` on 90d, `day`
  otherwise. Percentile colours mirror severity: p50
  foreground, p90 amber, p95 red, so tail regressions read at a
  glance.
- **Slowest Tests** — histogram bin width auto-picked from the
  window's `max(durationMs)` rounded to a nice step
  (100ms / 500ms / 1s / 2s / 5s / 10s / …) and inlined via
  `sql.raw`. The bottlenecks CTE does p95 picking + latest-meta
  picking + fail/flaky counting in one pass; pagination keeps
  Worker payload at 20 rows regardless of suite size. The
  7d sparkline query is batched across the current page's
  testIds (≤140 rows).
