# 2026-07-05 — Opt-in SQLCommenter query tags at the raw-SQL boundary

## What changed

Added opt-in SQLCommenter query tagging so PlanetScale Query Insights can attribute raw queries to a feature/deploy (and, later, so Postgres Traffic Control can match them). The dashboard talks to Postgres through Drizzle over node-postgres (`void/db`), which emits anonymous SQL — Insights could group by pattern but not by code path. Drizzle has no first-party SQLCommenter package, so tagging is **opt-in at the raw-SQL boundary** (`runRows`/`runRow`): a call passes `QueryTags` and the helper appends a `key='value'` block comment to the statement. Builder queries (`db.select()…`) are intentionally left untagged — they're cheap indexed lookups with no clean comment hook; the value is in attributing the heavy raw analytics/reporting reads.

This came out of a PlanetScale best-practices assessment (recommendations APP-1 + APP-2). The complementary agent-operating-model and Hyperdrive-runbook doc changes from that assessment are tracked separately.

## Details

| File                                                       | Change                                                                                                                                                                                                                                                                                                                                                         |
| ---------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `apps/dashboard/src/lib/db/query-tags.ts`                  | **New.** Pure SQLCommenter renderer (no `void/db`/`void/env` imports, so unit-testable): `QueryTags` type, `renderSqlCommenter(record)` (keys sorted; values percent-encoded so a `'` or `*/` can't break the comment/SQL string; `""` when empty), `buildTagComment(tags)` merging the baseline `application`/`service`/`environment`/`source`/`release_sha`. |
| `apps/dashboard/src/lib/db-run.ts`                         | `runRows`/`runRow` gained an optional `tags?: QueryTags`; a private `withTags` appends the comment (SQLCommenter convention) via `sql\`${query} ${sql.raw(comment)}\``. No tags → the query is returned unchanged, so untagged call sites are byte-for-byte unaffected. Re-exports `QueryTags`.                                                                |
| `apps/dashboard/src/lib/owners-repo.ts`                    | `latestFilePerTestId` raw read tagged `{ feature: "test-owners" }`.                                                                                                                                                                                                                                                                                            |
| `apps/dashboard/src/lib/monitors/http/uptime-analytics.ts` | Both raw reads (`httpResponseTimeBuckets`, `httpUptimeWindows`) tagged `{ feature: "monitor-uptime" }`.                                                                                                                                                                                                                                                        |
| `apps/dashboard/src/__tests__/query-tags.test.ts`          | **New.** 7 unit tests: key sorting, undefined/empty filtering, empty→`""`, encoding of `'`/`*`/`/`, route-template normalization, baseline + per-call override, `release_sha` omitted when unset.                                                                                                                                                              |

**`release_sha` source (APP-2):** read from a build-time `VITE_RELEASE_SHA` (inlined by Vite, like the existing `VITE_IS_DEV_SERVER`) — set `VITE_RELEASE_SHA=$(git rev-parse --short HEAD)` in the build/deploy step; omitted when unset. Build-time channel, not `void/env`, because the SHA is a property of the build, not request-time config.
