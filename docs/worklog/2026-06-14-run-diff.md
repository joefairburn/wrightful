# 2026-06-14 — Run-to-run comparison / diff (roadmap 2.4)

## What changed

Added a pure server-side set-diff between two runs' test results, keyed by the
stable `testId`. A run-detail page now links to a **Compare** view that diffs
the run against a baseline (by default the most recent passing run on the same
branch) and groups the results into Newly Failed / Newly Passed / Still Failing
/ Flaky changes / Added / Removed, each with per-test duration deltas.

This is a **pure read-path feature — no schema changes, no migrations.**
Everything reads existing columns (`testResults.{testId,status,durationMs,
retryCount}` and `runs.{id,status,branch,commitSha,commitMessage,createdAt}`).

## Diff status taxonomy

The diff classifies each `testId` from the vantage of "did this test get worse
or better between base and head". Each row gets a coarse verdict:

| Verdict   | Statuses                            |
| --------- | ----------------------------------- |
| `failing` | `failed`, `timedout`, `interrupted` |
| `passing` | `passed`, `flaky`                   |
| `other`   | `skipped`, `queued`, unknown        |

Notes on the choices (documented in `run-diff.ts`):

- **`interrupted` counts as failing here**, even though `@/lib/status` groups it
  with `flaky` for the UI counters. For a run-to-run diff, an interrupted test
  is a genuine non-pass (it didn't complete cleanly), so a `passed → interrupted`
  transition should surface as a regression. This is a deliberate, documented
  divergence from `statusGroupKey`.
- **`flaky` is `passing`** for the pass/fail verdict (the test ultimately ended
  green), but flaky-ness is surfaced separately via `flakyDeltas`.
- **`other` (skipped/queued) is neither**: `passed → skipped` is NOT a
  newlyFailed, and `skipped → passed` is NOT a newlyPassed.

Bucketing rules (a testId lands in at most one of the first three buckets, and
may _also_ appear in `flakyDeltas`):

- **`newlyFailed`** — passing/other/absent in base → failing in head (most
  important; rendered first).
- **`newlyPassed`** — failing in base → passing in head.
- **`stillFailing`** — failing in both runs.
- **`flakyDeltas`** — flaky verdict (`status === "flaky" OR retryCount > 0`)
  flipped, OR `retryCount` changed, for a test that is NOT already a pass/fail
  status change (avoids double-reporting).
- **`addedTests`** — present in head, absent from base.
- **`removedTests`** — present in base, absent from head.

`diffRuns` is **100% pure**: deterministic in its two inputs, no DB/clock/globals.
Output is emitted with testIds sorted, so re-running over the same inputs yields
byte-identical output. Per-test `durationDeltaMs = head.durationMs -
base.durationMs` is included on every status-change row.

## Base-run resolution

`resolveBaseRun(scope, head)` picks the natural baseline: the most recent run on
the **same branch** with `status = 'passed'`, created **before** the head,
excluding the head itself and any later runs. Two guards prevent self-comparison:
`lt(createdAt, head.createdAt)` AND `ne(id, head.id)` (belt-and-braces against a
same-second tie). Ordered `createdAt DESC LIMIT 1`, served by the existing
`runs_project_branch_created_at_idx` `(projectId, branch, createdAt)`. Returns
`null` when the head has no branch or there's no prior passing run.

The page/route also accept `?base=<runId>` to override the baseline; it's
validated through the project-scoped `loadDiffRunRef` (a foreign/missing id
degrades to the empty state, not a 404).

## Patterns followed

- Loader uses `requireTenantContext(c)`; all queries scope by `scope.projectId`
  (testResults reads) and `(teamId, projectId)` via the blessed `runScopeWhere`
  / `runByIdWhere` (runs reads). Logical tenancy — a leaked runId can't read
  another project's rows.
- JSON route uses `resolveTenantApiScope(c)` + the leak-safe 404, exactly like
  the sibling `summary.ts`, with `Cache-Control: private, max-age=30`.
- `ui/table`, `ui/badge`, `ui/empty`, `StatusGlyph`, `@/lib/status`,
  `time-format`, `cn()`, and `<Link>` (never `<a>`) for internal nav. Page is
  RSC; no `"use client"`. The base selector re-runs the loader on navigation
  (the diff is server-computed), so a `<Link>` list is the right primitive.
- Test follows the `void/db`-stub chainable-mock idiom from
  `quarantine-repo.test.ts` / `owners-repo.test.ts`.

## Adversarial review + fixes

Reviewed across four dimensions (diff-logic, tenant-isolation, base-resolution,
UI). Tenant isolation came back clean — `loadRunTestStatuses` is scoped by
`projectId AND runId`, `resolveBaseRun`/`loadDiffRunRef` are project-scoped, and
the `?base=` id is validated via the project-scoped lookup before its rows are
diffed in. Nine findings were confirmed (1 high, 3 medium, 5 low) and all fixed:

- **(high) `failing→other` mislabeled as a flaky change.** A test that failed
  after retries (`retryCount>0`) then stopped running (`failed→skipped`) fell
  through to the flaky-delta block and was reported as "flaky changed". The
  flaky-delta block now fires ONLY when BOTH rows' verdict is `passing`
  (passed/flaky) — closing both this and the `other→other` retry-noise case.
- **(low) `other→other` retry-count change** (e.g. `skipped→skipped` rc 0→1)
  emitted a spurious flaky delta — same root cause, same fix.
- **(low) misleading duration delta** for a non-running base (`skipped→failed`
  showed the head's full duration as a red regression). `StatusChange.durationDeltaMs`
  is now `number | null` — null when either side didn't run — and renders as "—".
- **(medium) same-second base excluded.** `createdAt` is epoch SECONDS, so a
  strict `lt(createdAt)` dropped a legitimate prior run created in the same
  wall-clock second as head (rapid CI re-fire). The boundary is now
  `or(createdAt < head, createdAt == head AND id < head.id)` using the ULID id
  (lexicographically time-ordered) to disambiguate within a second.
- **(medium) non-deterministic base pick.** Added `ORDER BY createdAt DESC, id
DESC` so ties on the second resolve deterministically to the genuinely most
  recent run.
- **(low) empty-string branch** was treated as a real branch (`eq(branch, "")`),
  grouping all branchless runs; `resolveBaseRun` and the base-candidate query now
  treat null/empty/whitespace alike (no base).
- **(low) UI**: centered both empty states (they hugged the header), hid the
  decorative `↗` glyph from AT + added an aria-label on the Compare link, and
  bottom-aligned it to the tab baseline.

Regression tests added for the diff-logic and base-resolution fixes.
